# StreamFusion: A Comet-style native accelerator for Flink SQL using Arroyo operators

Research notes. The project: accelerate Flink **SQL** by offloading supported relational/streaming
operators to a native Arrow/Rust engine built from **Arroyo's** operators, on the same architectural
model as **DataFusion Comet** for Spark — i.e. *data plane in Rust/Arrow, coordination stays in the
JVM (Flink)*. Scope is explicitly **Flink SQL**, not the DataStream API; UDFs are deferred.

All file:line references are against local checkouts:
- Flink: `~/data/flink` (post-2.0, `pre-apache-rename`)
- Arroyo: `~/data/arroyo` (master, ~v0.8.0+449)
- Comet: `~/data/datafusion-comet` (Spark Connect 4.0.1 era fork)

---

## 0. Executive summary

- **Feasible, but only for the SQL/relational subset** — the same shape as Comet. The clean
  "Java coordination / Rust data" seam exists at the Flink **SQL planner / ExecNode** layer, not for
  arbitrary DataStream jobs.
- **Arroyo is the right operator source**: it is Arrow-native, DataFusion-based, and — critically —
  uses **aligned Chandy-Lamport checkpointing with in-band barriers**, exactly like Flink. Its
  operators are *liftable* (Apache-2.0/MIT licensed) but **not** a drop-in library; they are coupled
  to Arroyo's columnar state backend and runtime.
- **No source fork required.** Like Comet (Spark = `provided` dependency + public extension API),
  you ship an add-on jar. **But** Flink exposes *no* public injection hook equivalent to Spark's
  `SparkSessionExtensions.injectColumnarRule`, so you reach internal seams (shadowed `PlannerFactory`
  / `ExecNodeGraphProcessor`, or `CalciteConfig.replaceStreamProgram`) and pay a heavier version-shim
  tax than Comet.
- **Two risks are quietly fatal and must be prototyped first**: (1) native off-heap **memory
  accounting** against Flink's managed-memory budget, and (2) the **threading/mailbox** bridge
  (Comet's "block the task thread" model is *illegal* in Flink). Both have clean designs below.
- **Two risks are silent-correctness, not fallback**: (3) the **changelog/`RowKind`** model, and
  (4) **watermark/timer** semantics (notably a window-close off-by-one).
- A clean, self-reinforcing **v1 envelope** falls out of all of this:
  > **Append-only, event-time windowed aggregation** (TUMBLE/HOP/SESSION), watermarks generated by
  > Flink, **no allowed lateness**, types in the safe intersection.
  This envelope dodges the changelog hazard entirely, keeps watermark alignment in Flink, needs only
  the off-by-one fix, and uses only types in the safe set.
- **#7 (does it actually *win* on perf?) is the unevaluated go/no-go.** Flink's bottleneck is often
  state I/O and network, not the CPU-bound expression eval that native vectorization accelerates, so
  the speedup ceiling is lower than Comet's. This needs a benchmark before deep investment.

## 1. Feasibility & the Comet analogy

Comet works because Spark gives three gifts: a **batch, pull-based** compute model; a **clean narrow
plan boundary** (post-Catalyst physical plan); and **stateless-per-partition** operators. Flink
violates all three for the DataStream API (record-at-a-time push; state interleaved with compute),
**but the SQL/Table layer restores them**: a Calcite-based plan with a bounded operator set and a
standardized changelog/watermark semantics. So the realistic project is *accelerate Flink SQL*, with
unsupported operators **falling back to Flink** exactly as Comet falls back to Spark.

Industry context: the more mature "native Flink SQL" bet is **Gluten + Velox** (same framework that
backs native Spark alongside Comet). Worth evaluating before building from Arroyo.

Flink SQL adoption (no clean public SQL-only-vs-UDF dataset exists): SQL is ~80%+ of streaming jobs
in China, **DataStream still dominant in US/EU**; managed platforms (Confluent Cloud, AWS MSF,
Ververica) are SQL-first and growing; Flink 2.x is strategically SQL-centric (removed DataSet API,
added Process Table Functions to bridge SQL↔DataStream). The high-value *expensive* jobs in Western
shops often sit in DataStream where this can't reach — so measure the offloadable fraction of the
actual target fleet.

## 2. Checkpointing — confirmed compatible (the easy half)

Arroyo uses **aligned Chandy-Lamport** barriers carried **in-band** in the data stream, just like
Flink:

```rust
// arroyo-types/src/lib.rs:182
pub enum ArrowMessage { Data(RecordBatch), Signal(SignalMessage) }
pub enum SignalMessage { Barrier(CheckpointBarrier), Watermark(Watermark), Stop, EndOfData } // :187
pub struct CheckpointBarrier { epoch: u32, min_epoch: u32, timestamp: SystemTime, then_stop: bool } // :500
```

- `CheckpointBarrier{epoch,timestamp,...}` is field-compatible with Flink's `CheckpointBarrier`.
- Multi-input alignment is a per-input counter (`CheckpointCounter`, `arroyo-operator/src/lib.rs:70`)
  — aligned/buffered, same as Flink's default.
- **Key consequence:** if Flink owns the runtime, you **drop** Arroyo's checkpoint coordination,
  source-side barrier injection, and `CheckpointCounter` entirely — Flink injects/aligns barriers and
  calls `snapshotState()`. The native operator just receives "snapshot epoch N." Coordination stays
  in Java, as desired.
- The operator trait ports cleanly: `ArrowOperator` (`arroyo-operator/src/operator.rs:1144`) —
  `process_batch` is the core; `handle_checkpoint`/`handle_watermark` default to no-op/pass-through.

**The real coupling is state, not barriers** — see §6.

## 3. Planner hook — no public front door (use Seam B, fork later)

Flink has **no** public injection API like `SparkSessionExtensions`. Options, increasing invasiveness:

1. **`CalciteConfig.replaceStreamProgram`** (`CalciteConfig.scala:60`; read by
   `StreamCommonSubGraphBasedOptimizer.scala:199`). The only *config-reachable* hook. Operates at the
   physical-RelNode layer. It's **replace, not inject** — you rebuild Flink's default
   `FlinkChainedProgram` from `@Internal` `FlinkStreamProgram`/`FlinkStreamRuleSets` and splice your
   phase in.
2. **Shadow the `"default"` `PlannerFactory`** (the discovery identifier is hardcoded:
   `PlannerFactoryUtil.java:48` → `PlannerFactory.DEFAULT_IDENTIFIER` = `"default"`,
   `PlannerFactory.java:42`). Ship a factory under `"default"` returning a `StreamPlanner` subclass
   that overrides `getExecNodeGraphProcessors` (which is **empty + not an SPI** for streaming:
   `StreamPlanner.scala:76`). This **`ExecNodeGraphProcessor`** layer is the closest analog to how
   Comet rewrites the physical `SparkPlan`, and the right place for operator-for-operator swaps +
   fallback policy. This is **"import + classpath/service shadow + subclass internals" — NOT a source
   fork.**
3. **Patch/rebuild `flink-table-planner`** — an actual fork. Avoid until reachability forces it.

**Decision: build on #2; migrate to #3 later if needed.** The work you build (`ExecNodeGraphProcessor`,
custom ExecNodes, their `translateToPlanInternal` → `Transformation`, the native operators) is
**identical** in both; only *how the processor is registered* changes. Fork only when a hook you need
is `private`/`final`/non-virtual.

### Fork vs import (confirmed)
Comet does **not** fork Spark: `spark.maven.scope=provided` (`pom.xml:74`); registers via
`class CometSparkSessionExtensions extends (SparkSessionExtensions => Unit)` calling `injectColumnar`
/ `injectQueryStagePrepRule` (`CometSparkSessionExtensions.scala:90-98`), activated by config
(`spark.sql.extensions=...`); absorbs version drift with **shims** (`org.apache.comet.shims.*`). You
replicate the *packaging* (provided dep + shim jar) fully; the difference is the entrypoint, hence the
heavier shim tax against `@Internal` planner classes.

## 4. Memory accounting (#1, showstopper-class) — copy Comet's mechanism

Comet's mechanism (the **Rust half is directly reusable**): DataFusion's `MemoryPool` trait is the
seam. Comet implements `CometUnifiedMemoryPool`/`CometFairMemoryPool`
(`native/core/src/execution/memory_pools/`) whose `try_grow`/`shrink` call back **via JNI** into a
JVM bridge that defers to Spark's budget:

```
DataFusion op → try_grow(n) → CometUnifiedMemoryPool.acquire_from_spark(n)
  → jni_call!(comet_task_memory_manager.acquire_memory(n))
    → CometTaskMemoryManager.acquireMemory(n)  [Java bridge]
      → Spark TaskMemoryManager.acquireExecutionMemory(n, consumer)
```
Grant `< n` → Rust returns `resources_err!` → DataFusion operator spills. Requires
`spark.memory.offHeap.enabled`; pool is per-task, shared across operators.

### Flink mapping (APIs verified present)
| Comet (Spark) | StreamFusion (Flink) | Evidence |
|---|---|---|
| `CometUnifiedMemoryPool` (DataFusion `MemoryPool`) | **reuse** as `FlinkManagedMemoryPool`, same trait + JNI callback | — |
| `CometTaskMemoryManager` (Java bridge) | `ArroyoFlinkMemoryManager` (Java bridge) | — |
| `TaskMemoryManager.acquireExecutionMemory/release` | `MemoryManager.reserveMemory(owner,size)` / `releaseMemory(owner,size)` | `MemoryManager.java:413,435` |
| `spark.memory.offHeap.{enabled,size}` | `taskmanager.memory.managed.{size,fraction}` | — |
| `memoryPool.fraction` | managed-memory weights via `ManagedMemoryUseCase.OPERATOR` + `computeMemorySize(fraction)` | `ManagedMemoryUseCase.java:29`, `MemoryManager.java:653` |
| TaskContext-provided manager | `getContainingTask().getEnvironment().getMemoryManager()` | `Environment.java:163` |

The model to mirror exactly is `MemoryManager.getSharedMemoryResourceForManagedMemory`
(`MemoryManager.java:514`) — it reserves managed memory (`:530`), returns an `OpaqueMemoryResource`,
releases on close (`:542`). **This is how RocksDB plugs its off-heap cache into Flink's budget.** Good
news: managed memory is the *right* shared budget and Flink already multiplexes competing off-heap
consumers — arguably cleaner than Comet.

### Three ways it's harder than Comet
1. **No cooperative cross-operator spill.** Spark's `acquireExecutionMemory` gives *partial* grants
   and asks peers to spill; Flink's `reserveMemory` is **binary** (reserve or throw). Denial is a hard
   failure unless your own operator spills.
2. **State is the dominant consumer**, not transient working memory (Comet's case). Large, long-lived
   streaming state (windows/joins) must also be reserved.
3. **Arroyo's streaming operators lack runtime disk-spill** (their state flushes to object store on
   *checkpoint*, not under back-pressure). The "denied → spill → continue" escape Comet relies on may
   not exist.

**Resolution — same decision as the state backend:** lean on Arroyo's **object-store / disaggregated
state** (bounded in-memory cache; aligns with Flink 2.x ForSt direction) so the pool bounds only
cache + working buffers (Comet's comfort zone). Commit to **resident** state and you inherit a spill
problem Flink won't help you negotiate.

## 5. Threading / mailbox (#2, showstopper-class) — Comet's bridge does NOT port

Three models, **opposite thread ownership**:

| | Execution | Thread ownership | Blocking |
|---|---|---|---|
| Spark (Comet host) | pull, batch | thread-per-task | **fine** |
| **Flink (host)** | push | **one mailbox thread / task** | **forbidden** (stalls barriers/timers/watermarks) |
| Arroyo (operators) | push, in-band barriers | **autonomous actor** — each op a `tokio::spawn` w/ own `select!` loop | async throughout |

Comet parks the Spark task thread in `block_on`/`blocking_recv` (`jni_api.rs:812`) on a **global
multi-thread `OnceLock<Runtime>`** and pulls one batch per JNI call. **Doing that on Flink's mailbox
thread would stall checkpointing for the whole task.** So Comet's threading model is non-transferable.
Also: Comet *flattens* async→sync pull because Spark is pull/batch; Arroyo is push-with-in-band-barriers
— **exactly like Flink** — so you must NOT flatten it.

### The design that fits (verified primitive: `MailboxExecutor.execute`/`yield`, `MailboxExecutor.java:136,197`)
1. **One shared multi-threaded tokio runtime per TaskManager** (copy Comet's `OnceLock<Runtime>`,
   `jni_api.rs:121`).
2. **Host an Arroyo *chain* (not a single operator) as one Flink operator**, running Arroyo's existing
   `operator_run_behavior` `select!` loop (`operator.rs:932`) as an autonomous task. The native chain
   is the contiguous offload unit (Arroyo's `ChainedOperator`). Re-enter Java only at chain ends
   (shuffles, sinks).
3. **Ingress (Flink→native)** in `processElement`: batch `RowData`→Arrow, push to the chain's input
   `BatchSender`. If full, **do not block** — loop with `mailboxExecutor.yield()` so barriers/timers
   still run (the `AsyncWaitOperator`/FLIP-27 pattern).
4. **Egress (native→Flink)**: native tail enqueues a **mailbox action** (`mailboxExecutor.execute`)
   that, *on the mailbox thread*, drains output and calls `output.collect`. Mandatory: `output.collect`
   and state access are mailbox-thread-confined — a tokio worker must never call them.
5. **Checkpoint**: Flink calls `snapshotState` on the mailbox thread → inject the barrier into the
   chain's **control channel** (Arroyo already has `ControlMessage::Checkpoint`) → reuse Arroyo's
   `run_checkpoint` → mailbox waits (yield-loop) for the ack, registers the snapshot as Flink state.

This **reuses Arroyo's `select!` loop, channels, `future_to_poll`, `BackendFlusher`, barrier handling
intact** — you match *thread ownership*, not execution model.

**Refinement (hybrid):** ~90% of Arroyo compute is "async only at the edges" (sync Arrow/DataFusion
kernels; await only on channels). Drive **stateless / synchronously-completing** ops (project/filter)
**Comet-style** (short `block_on` in `processElement`). Reserve the actor + mailbox bridge for
**time-/future-driven** ops (windowed aggs spawn background DataFusion futures via `future_to_poll`,
`tumbling_aggregating_window.rs:394`; joins; lookup). **Discriminator:** *does the operator make
progress when no input arrives?* yes → actor; no → synchronous drive.

**Top risk: mailbox ↔ native-backpressure deadlock** (native blocked waiting for mailbox to drain
output; mailbox blocked in `processElement` waiting for native to accept input). The yield-on-both-waits
handshake prevents it but must be designed and stress/model-checked deliberately — peer to memory in
severity.

## 6. Changelog / `RowKind` (#3, silent-correctness) — the decision lives in Flink's planner

Flink decides **per node** whether to emit retractions, via three trait passes
(`FlinkChangelogModeInferenceProgram`):
- **`ModifyKindSet`** — what changes a node produces (I/U/D).
- **`UpdateKindTrait`** — `ONLY_UPDATE_AFTER` (upsert) vs `BEFORE_AND_AFTER` (retract), decided by
  whether an **upsert key** exists and the downstream/sink requirement.
- **`DeleteKindTrait`** — key-only vs full-row deletes.

The planner stamps a flag (e.g. `StreamExecGroupAggregate.generateUpdateBefore`) and the runtime obeys
it (`GroupAggHelper`: emit `-U`+`+U` iff the flag, else `+U` alone; `-D` when count→0). RowKinds:
`+I / -U / +U / -D` (`RowKind.java`), carried as metadata on `RowData` (not a column).
`ChangelogMode` (`ChangelogMode.java`): `insertOnly` / `upsert(keyOnlyDeletes)` / `all`.

**Implications for StreamFusion:**
- Offloaded operators **must honor Flink's planner flags**, not Arroyo's own retraction decisions.
  Since you discard `arroyo-planner` (you translate Flink's plan directly), you control this — the work
  is making Arroyo's *operators* honor an externally-supplied changelog flag.
- **Representational width gap (the deepest hazard):** Flink has 4 RowKinds; Arroyo's runtime model is
  a **2-way** insert/retract (`is_retract`/`_updating`). For **upsert sinks** the `-U` vs `-D`
  distinction is critical (upsert sinks *drop* `-U` but *act on* `-D`); a 2-way encoding can't recover
  it → silent corruption. For pure **retract** sinks the +/- sign suffices. So **verify/extend Arroyo's
  batch encoding to carry the 4 RowKinds** before offloading anything feeding an upsert sink.
- **Boundary contract:** when you cut back to Flink, the emitted stream must match the
  `ChangelogMode`/`UpdateKind` trait Flink assigned to *that edge*.

**Mitigation that dissolves all of the above for v1: offload only append-only (insert-only) streams.**
A windowed aggregation over an append-only source emits **INSERT-only** — no flags, no 2-vs-4 gap, no
boundary trait. Everything changelog-bearing (non-windowed `GROUP BY`, CDC sources, upsert sinks) stays
on Flink.

## 7. Watermarks / timers (#4, silent-correctness) — compatible, two concrete traps

| Aspect | Verdict |
|---|---|
| Watermark propagation + multi-input **min** | **Identical** (`StatusWatermarkValve` ↔ `WatermarkHolder.update_watermark`, `context.rs`) |
| **Idleness** | Compatible — Arroyo `Watermark::Idle` enum (`arroyo-types/src/lib.rs:175`) vs Flink `WatermarkStatus.IDLE`; mechanical bridge |
| **Alignment (FLIP-182)** | Arroyo missing — **irrelevant if Flink generates watermarks** (it's a source-side throttle applied upstream of your operator) |
| **Window triggering** | Same intent, **off-by-one** — see trap 1 |
| **Late data** | Diverges **only if allowed-lateness configured** — see trap 2 |

1. **Window-close off-by-one.** Flink fires when `timer.ts <= watermark`
   (`InternalTimerServiceImpl.advanceWatermark`); Arroyo uses strict `<` on `bin_start`
   (`tumbling_aggregating_window.rs:321`), emitting a window *one watermark later*. Different results
   for events exactly on a window edge. Fixable (`<=` vs `<`) but a classic silent parity failure.
2. **Late data.** Arroyo **drops** records behind the watermark (`session_aggregating_window.rs:285`).
   Flink's **default** also drops, so default-config jobs match; **allowed-lateness** jobs diverge
   (Flink emits late updates). Gate offload on "no allowed lateness" (also excluded by append-only
   scope, since lateness produces retractions).

Event-time timers: Arroyo has **no `InternalTimerService`** — it uses watermark-driven flushing. The
window operators you port already use this model, so you inherit a working approach; just match firing
semantics (the off-by-one + order). Keep **watermark generation in Flink** so alignment is handled
upstream.

## 8. Type mapping (#5) — mostly solved; Flink already has an Arrow bridge

Flink ships `ArrowUtils` / `LogicalTypeToArrowTypeConverter`
(`flink-python/.../table/runtime/arrow/ArrowUtils.java`) for the PyFlink vectorized-UDF path — a
proven `RowData↔Arrow` mapping you can lift. Offloadable type surface =
**{ArrowUtils supports} ∩ {Arroyo/DataFusion accepts}**:

| Type | Flink→Arrow | Arroyo | Verdict |
|---|---|---|---|
| numerics, bool, string→Utf8, varbinary, date | ✓ | ✓ | **safe** |
| DECIMAL(p,s) | ✓ Decimal128 | ✓ | safe **if** scale≥0, p≤38 (Arrow has no negative scale; Flink allows) |
| TIMESTAMP without TZ | ✓ `Timestamp(unit,null)` | ✓ (precision 0/3/6/9 only) | safe; **gate precision** |
| TIMESTAMP with local TZ (LTZ) | ✓ `Timestamp(unit,null)` UTC instant | ✓ (UTC instant) | **safe for windowing** |
| TIME | ✓ precision-mapped | ✗ `Time64(Nano)` only | gate to Nano or fall back |
| ARRAY / ROW | ✓ List / Struct | ✓ if children supported | safe, recursive |
| MAP | ✓ | ⚠ planner won't ingest; operators can carry | passable where merely carried |
| TIMESTAMP **with** TZ, INTERVAL, MULTISET, RAW, STRUCTURED, VARIANT | ✗ (bridge throws) | ✗ | **hard fallback** |

Notes:
- **LTZ is the usual Flink rowtime type**; both sides map it to a zone-less UTC instant, correct for
  windowing (boundaries computed on the instant). Divergence only in session-local
  formatting/extraction, absent in pure windowed aggregation.
- **Arroyo injects a special `_timestamp` column** (`schemas.rs:add_timestamp_field`); map Flink's
  rowtime attribute → `_timestamp` at the boundary.
- **What to build:** replicate Comet's recursive `DataTypeSupport.isTypeSupported`
  (`spark/.../DataTypeSupport.scala`) — accumulates fallback reasons, surfaces in EXPLAIN, composes
  with operator-granularity fallback.
- The v1 windowed-agg type set (numerics, string, decimal, LTZ rowtime) is entirely in the safe set.

## 9. Parity testing (#6) — Comet's stack transfers; streaming adds one hard wrinkle

Comet's six layers (all in `spark/src/test/.../CometTestBase.scala` + `CometPlanChecker.scala` +
`CometFuzzTestBase.scala`):
1. **Reuse the host's own SQL suite** (Comet diff-patches Spark to force Comet on; thousands of tests).
2. **`checkSparkAnswerAndOperator`** — run native-off (baseline) vs native-on, compare results, **AND
   assert the plan is fully native** (silent fallback = worthless test; `findFirstNonCometOperator`).
3. **Float tolerance + NaN/Inf special-casing + ordering normalization.**
4. **Fuzzing** — random schemas + edge-case data (min/max/NaN/Inf/null/decimal-overflow).
5. **Incompatibility allowlist** — `spark.comet.expression.<Expr>.allowIncompatible` documents known
   DataFusion-vs-Spark diffs. **You'll need this heavily** (DataFusion vs Flink expression semantics:
   decimal rounding, string casts, regex).
6. **`.sql` golden files** with modes (`CheckCoverageAndAnswer`/`ExpectFallback`/`ExpectError`).

**Streaming wrinkle Comet never faced:**
- **Nondeterministic timing.** Drive both impls in **lockstep** with Flink's deterministic harnesses
  (`KeyedOneInputStreamOperatorTestHarness`: push elements, `processWatermark()`, snapshot). The single
  most important streaming-parity tool.
- **Changelog-equivalence ≠ row-equivalence.** Flink may emit `+I` then `-U/+U` while native emits `+I`
  at the final value — both correct. Correct comparison: **fold both changelog streams into a keyed
  table and compare materialized final state**, not the emission sequence.

**v1 again simplifies it:** append-only output has no retractions → the changelog *is* the final state
(one `+I` per closed window) → compare the **multiset of emitted rows** (Comet's batch diff works); the
only streaming-specific machinery needed is the **deterministic watermark harness**. Defer the
changelog-folding comparator to v2.

**v1 parity harness to build:** identical input + scripted watermarks → stock Flink SQL vs native path
via `*StreamOperatorTestHarness`; multiset comparison w/ float tolerance; operator-coverage assertion
(the `findFirstNonCometOperator` analog over the Flink ExecNode/Transformation graph); edge-case
generators + DataFusion-vs-Flink incompat allowlist; **boundary-timestamp fuzzing** specifically (where
the §7 off-by-one hides).

## 10. Operator & connector compatibility (reference)

### Flink SQL operators (~57 `StreamExec*` nodes), mapped to Arroyo
**Direct map (the offloadable core — common analytics streaming SQL):** Calc/Union/Values/Expand
(DataFusion); GroupAggregate↔IncrementalAggregatingFunc; Window aggregates TUMBLE/HOP/SESSION↔
Tumbling/Sliding/Session; OverAggregate↔WindowFunctionOperator; Interval/Window join↔JoinWithExpiration
(⚠ inner/left only); regular Join↔InstantJoin (⚠ inner/left); LookupJoin↔LookupJoinOperator; AsyncCalc↔
AsyncUdfOperator; WatermarkAssigner↔WatermarkGeneratorOperator.

**Gaps → Flink fallback:** MATCH_RECOGNIZE/CEP (no Arroyo equivalent); TemporalJoin (versioned);
FULL/RIGHT/OUTER/ANTI/SEMI joins (Arroyo inner/left only); **CUMULATE** windows; TopN/Rank-with-retraction,
ORDER BY, LIMIT, Sort (Arroyo planner explicitly rejects ORDER BY/LIMIT); Deduplicate; MultiJoin/DeltaJoin;
GroupTableAggregate; ML_PREDICT/VECTOR_SEARCH/PTF; Python ops.

Arroyo operators live in `arroyo-worker/src/arrow/`. You **reuse the operator kernels, not
`arroyo-planner`** — you write a Flink-plan→Arroyo-operator translator (the `QueryPlanSerde` analog).

### Connectors — NOT ported (Flink owns I/O)
In the accelerator, sources/sinks/checkpointing stay in Flink, so you **inherit Flink's entire connector
+ format ecosystem for free**. Arroyo's connector list only matters if you ever keep I/O in Rust (see
below). Genuine overlap is essentially **Kafka + object-store filesystem**. Flink's strength:
DB/warehouse/cloud sinks (JDBC, ES/OpenSearch, Hive, Kinesis…). Arroyo's strength: lightweight/edge
sources (SSE, WebSocket, MQTT, NATS, RabbitMQ, HTTP-poll) + Delta/Iceberg sinks. Format gaps in Arroyo:
no CSV/ORC, Protobuf input-only, Parquet output-only, only Debezium-JSON CDC (vs Flink's
Debezium/Canal/Maxwell/OGG).

## 11. Optional later: connectors in Rust (end-to-end Arrow)

Keeping the Kafka source/sink in Rust (Arroyo's) yields an **end-to-end Arrow data plane** (no
JVM deserialize + transcode), but pulls *more* runtime contract into Rust:
- **Source (easy):** Kafka offsets are just state → snapshot via the same Flink-barrier→Rust-snapshot
  bridge (offsets = state). Lose Flink's FLIP-27 split enumeration/rescaling.
- **Sink (hard, exactly-once):** Flink's two-phase commit pre-commits on barrier, **commits on
  `notifyCheckpointComplete`**. Bridge that to Arroyo's commit phase (`is_committing`/`handle_commit`).
  Highest correctness risk.
- Only pays off across a **contiguous native span** (Rust-source → native ops → Rust-sink, no Java UDF
  in between).
- **Sequencing:** Flink Java connectors first (correctness + ecosystem) → Rust Kafka source → Rust
  exactly-once sink last.

## 12. UDFs (deferred — recorded for later)

Approach validated against the Comet fork's `CometScalaUDF`/`CometBatchKernelCodegen`: a UDF is **not**
translated to Rust. Native execution **calls back into the JVM** at *batch* granularity; the UDF runs
on the JVM over Arrow buffers (Janino-compiled kernel reads Arrow vectors directly, no per-row
columnar→row transition). Two wins: (1) keeps the plan native at **expression granularity** (vs the
whole operator falling back); (2) codegen makes the in-tunnel eval fast. For Flink, the equivalent is a
JVM-side batch loop over Arrow feeding `ScalarFunction.eval`; the unavoidable cost is per-row
Arrow↔Catalyst-value marshalling (free for primitives). **Bridges the Arroyo-expects-Rust-UDFs vs
Flink-has-Java-UDFs mismatch.** Out of scope for v1 (SQL built-ins only).

1. **Prove #7** — append-only windowed-agg benchmark; is the win real on a state-bound engine?
2. **De-risk #1 (memory) & #2 (threading)** — the two quietly-fatal mechanisms; build the
   `FlinkManagedMemoryPool` + JNI bridge and the mailbox ↔ actor handshake (with deadlock stress test).
3. **Vertical slice**: one tumbling windowed aggregation, end-to-end, offloaded via a shadowed
   `PlannerFactory` + `ExecNodeGraphProcessor`, with the barrier→Rust state-snapshot bridge, checkpoint
   + recover under Flink.
4. **Parity harness** (deterministic watermarks + multiset diff + operator-coverage assertion +
   boundary-timestamp fuzzing); fix the §7 window-close off-by-one.
5. Commit to **disaggregated/cached state** (resolves the memory-spill gap).
6. Expand operator coverage within the append-only envelope; defer changelog-bearing ops, UDFs, and
   Rust connectors to v2.
