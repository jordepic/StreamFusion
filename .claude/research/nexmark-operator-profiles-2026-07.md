# Nexmark operator profiling round — 2026-07-04

Differential CPU profiles (async-profiler 4.4, 35s steady-state samples) of **every accelerated
Nexmark query, native vs stock Flink**, on the generator profile loop
(`SF_PROFILE=true`, `NexmarkMatrixBenchmark#generatorNativeProfileLoop`, release+mimalloc build,
Apple M1 Max, 500K events/iteration). Raw recordings, collapsed stacks, and flame-graph HTML per
query/engine live in `.claude/profiles/2026-07-04/` (not committed; regenerate with the loop —
method below). Numbers are **CPU samples per benchmark iteration** (sampling rate equal across
runs, so per-iteration samples ∝ CPU cost per unit of work).

**Why the generator rung:** it is the purpose-built loop for operator flame graphs — no Kafka, no
decode; transpose → native island → transpose. The Kafka JSON rung's profile is dominated by
consume+decode, already profiled and optimized separately (divergences/19). Operator-side stacks
are identical across rungs.

**Method caveats:**
- The loop submits a fresh job per iteration, so Flink codegen'd classes are recompiled every run —
  the JIT share of "housekeeping" in the *flink* column is partly a harness artifact (measured: q16-flink
  17.2 JIT samples/iter vs 1.6 GC). GC, by contrast, is a real steady-state cost.
- Combined-loop numbers understate native slightly vs the matrix's fresh-JVM runs.

## The target, grounded (Alibaba's "5-10x")

- The 5-10x claim = **Flash** (Alibaba's vectorized C++ Flink-compatible engine; sold as Ververica
  VERA-X): 8.4x aggregate at 100M events / 5.7x at 200M vs Flink 1.19. **No per-query breakdown is
  published.** No academic paper exists; other Alibaba material says "3-4x".
- The only public per-query table is Alibaba's managed **Java** engine (mini-batch 2s + distinct-agg
  split enabled): **3.24x aggregate**, with q9 8.7x, q19 6.2x, q18 6.0x, q20 5.8x, q15 4.3x …
  q5 1.02x (worst). I.e. the big multipliers are exactly the changelog/stateful queries, won by
  **amortizing per-key state work over batches** — and their worst query is the one vectorization
  doesn't help (hopping windows).
- Flash's credited techniques — vectorized operators on ~1000-row batches, native rewritten scalar
  functions, arena-pooled batched state access, whole-plan-native-or-fallback — are this project's
  architecture. Their numbers are cluster runs with columnar internals end-to-end; our generator
  rung deliberately pays a rowwise perimeter both ways. **The honest comparison for "5-10x" is our
  columnar-source rungs (Parquet 1.2-5.1x, native-Kafka 1.4-4.8x today), not the generator floor.**

## Cross-query differential (samples/iteration)

See the summary table generated by `summary_matrix.py` (reproduced at the bottom). Reading:

- **The rowwise perimeter dominates everything on the generator rung.** For q0 the native run's
  15 samples/iter split: datagen 2.7 + Row→RowData conversion + Flink task/mailbox 4.8 + entry/exit
  transpose 3.2 + island 0.5. Even a zero-cost native island would cap q0 near ~1.5x. 5-10x is not
  reachable over this perimeter for stateless queries; it is reachable for stateful ones (Flink's
  operator+GC cost is huge there) and for columnar-source deployments.
- **Flink's stateful queries are GC-bound:** q18-flink pays 34 GC samples/iter and q9-flink 23
  (native: ~1). This is the structural win of arrow-native state — and why Alibaba's Java tier
  needed mini-batch for its 6-9x on q9/q18/q19.

## Systemic findings (the levers), ranked by total headroom

1. **[shipped 2026-07-04: q3 +17%, q9 +9%, q20 +6%]** **Disabled sub-plan reuse forks the rowwise
   prefix — every multi-view query converts the source stream twice.** Measured: `RowRowConverter` samples/iter native-vs-flink — q0/q19
   (single view) 2.2/2.2 and 2.5/2.5; q3 4.8/2.3, q4 4.7/2.5, q5 4.6/2.2, q7 5.0/2.2, q8 4.6/2.2,
   q9 4.4/2.5, q20 4.6/2.2 — an exactly-2x tax on 7+ queries (~10% of q3's whole job; datagen
   itself is NOT doubled — the fork is after the source function, at the conversion node).
   The reuse ban (`NativePlanner.install`) exists because an Arrow batch is consumed once
   (zero-copy hand-off) — but that invariant only concerns **ArrowBatch-typed** edges. Reuse of
   the **rowwise prefix** (source + Row→RowData conversion) is safe. Fix: scope the reuse ban to
   columnar sub-plans (or re-enable reuse and insert a defensive batch-share/clone only where an
   Arrow edge fans out). On Kafka rungs the same fork duplicates consume+decode — worth verifying.
2. **The entry transpose (Row→Arrow) is the single biggest native-side line on the changelog
   queries, and it tracks string volume.** q9 9.5, q20 9.1, q18 6.6, q19 6.2 samples/iter (exit
   transpose is negligible except q19's 4.8, on the amplified rank cascade). q0 pays only 2.8
   because projection pushdown drops url/channel. Half of q9/q20's cost is the forked prefix
   (lever 1). The rest: per-row, per-field `ArrowFieldWriter` writes; strings dominate.
   `BinaryStringData` already holds UTF-8 bytes — a bulk bytes→builder fast path (and
   column-major write batching) is the ticket-20 "column-vectorized rewrite" with profile evidence.
3. **Streaming Top-N emit is 72% arrow-row DECODE.** q19's `pushTopNRanker` (2119 samples):
   `decode_binary` 621 + `decode_fixed` 328 + `decoded_len` 192 + `decode_nulls` 104 + memmove 281.
   Every emitted row (and the rank cascade amplifies output) is decoded from stored arrow-row bytes
   into columns one row at a time. References: Proton stores join/agg state as **columnar blocks +
   {block, row} refs** and emits by `take` from blocks; RisingWave's Top-N keeps a three-zone cache
   and emits **net diffs** (`TopNStaging` cancels intra-batch shift churn before emission).
   Both directions apply: (a) stage the shift cascade and emit the net change per batch,
   (b) keep Top-N state (or a parallel copy of the hot window) in a form that can be `take`n
   rather than decoded per row.
4. **[shipped 2026-07-04]** **Keep-last dedup (q18) spends 35% of its island in stdlib SipHash.** `sip::Hasher::write` 381 +
   `hash_one` 215 of 1697 island samples, plus `reserve_rehash` 53. The dedup key map never got the
   ahash + arrow-row-key swap the aggregators got (ticket 20's known survivor list: dedup, OVER
   partitions, Top-N bookkeeping, exchange split). Profile says the swap pays for dedup; do the
   others while there.
5. **Multi-DISTINCT group aggregate churns ScalarValue** (q16 island 30.8 samples/iter, q15 18.6,
   q17 19.4): `drop_in_place<ScalarValue>` + `clone` + `hash` + `try_from_array` + allocator traffic
   ≈ half the operator. RisingWave's model: dedup by **mutating the batch's visibility bitmap** with
   one `Box<[i64]>` ref-count array per distinct key serving ALL distinct calls on that column;
   Proton's model: dirty-flag headers + emit walks only changed groups. Ties into ticket 41
   (`IncrementalGroupAggregate` under mini-batch is what Flink plans distinct aggs to).
6. **Lookup join upcall pays a defensive `RowDataSerializer.copy` per looked-up row** (q13: ~400 of
   1471 lookup-path samples, plus `ensureMaterialized`/`StringUTF16.checkIndex`). This is ticket
   40's columnar-assembly item, now with numbers. Arroyo also batches lookups with in-batch key
   dedup — one connector call per batch of misses.
7. **Updating-join state store** (q3/q20/q9; islands 4-7 samples/iter in the joiner): per-row
   `OwnedRow` pairs into nested HashMaps. References agree on the fix: Proton's
   `RefCountDataBlockList` (state = few big columnar blocks; hash table maps key → {block, row}
   refs in arena nodes of 7; GC block-granular by refcount; `RowRefListMultiple` for retract
   erase) — i.e. our state becomes append-to-Arrow-block + offset index; RisingWave's
   `CompactedRow` (one `Bytes` blob) + zero-alloc view composition (`Chain`/`Project`) is the
   lighter-touch variant. Combined with lever 1+2 this is what closes q3/q20.
8. **GROUP BY emit can skip clean groups.** Proton's `TrackingUpdates` dirty-flag header per group
   state (+ separate bulk-wiped retract arena, + "many_updated_keys" fast path iterating only
   touched keys) makes finalize O(changed), not O(table). Arroyo adds retract==append no-op
   suppression. Our group agg emits per flush from full state scans (q4/q15/q16/q17).
9. **[compile-cache shipped 2026-07-04; string hand-off remains]** **The q21 byte-parity upcall
   re-materializes and maybe re-compiles.** q21-native leaves:
   `String.<init>` 5.1% + `StringLatin1.toLowerCase` 4.3% + `StringUTF16.checkIndex` 3.9% (the
   upcall builds a Java `String` per row from Arrow, runs regex/case, writes back through
   `ArrowFieldWriter`), and — suspicious — `java.util.regex.Pattern.compile` at 2.3% *self* time:
   the constant pattern looks like it is being recompiled per batch (or worse) somewhere in the
   upcall path instead of cached in the registered function instance. Verify and cache; that plus
   bulk string hand-off is q21's 0.75x deficit (its opt-in pure-Rust path already runs 1.56x).
10. **q23 confirms the join story at three-way scale**: updating_join 13.3 samples/iter (the
   largest joiner island), and stock Flink pays 62.6 housekeeping samples/iter (mostly GC) — the
   biggest GC number in the sweep. Levers 1+2+7 all apply.

## Reference-technique shortlist not yet tied to a measured hot spot

- Arroyo: per-bin background partial aggregation driven from the operator's select loop; paned/
  tiered sliding-window aggregation (q5 — hop windows; Alibaba's worst query too); intrusive
  linked-list TTL cache (`UpdatingCache`) for O(1) touch / O(evicted) expiry; row-count-bounded
  backpressure channel.
- RisingWave: precomputed-hash keys (`PrecomputedBuildHasher`), stack-allocated `FixedSizeKey<N>`;
  adaptive Vec(≤4)→BTreeMap per-key join entry; in-place changelog compaction
  (`StreamChunkCompactor`, `eliminate_adjacent_noop_update`).
- Proton: time-bucket hash tables with per-bucket arenas (window GC = drop bucket + arena, O(bucket));
  watermark as chunk metadata (already ours); saved-hash cells.

## Where 5-10x stands

Per-query native/flink CPU-per-iteration ratios today (generator): stateless ~1.1-1.6x capped by the
shared rowwise perimeter; stateful 1.0-2.5x with the levers above unshipped. The path to a 5-10x
class result: (a) land levers 1-8 (they compound: q19's operator+transpose is ~2/3 of its native
CPU; q3/q20's fork+transpose+state-store likewise); (b) measure the headline on the columnar rungs
(Parquet already 2-5.1x, native-Kafka 1.4-4.8x) — which is what Flash-class engines actually
benchmark; (c) mini-batch/dirty-key amortization for the changelog family (Alibaba's own Java
engine gets its 6-9x there from mini-batch). The rowwise-generator floor cannot reach 5x for
pass-through queries — the source+runtime perimeter alone caps it — and that matches Flash's
positioning (whole-plan columnar, columnar exchanges).

## Reproduce

```bash
# one query, native side (flink side: -Dprofile.native=false)
SF_BENCHMARK=true SF_PROFILE=true mvn test -Pbench \
  -Dnative.cargo.args="build --release --features mimalloc" \
  -Dtest='NexmarkMatrixBenchmark#generatorNativeProfileLoop' \
  -Dprofile.query=q19 -Dprofile.seconds=75 &
asprof -d 35 -e cpu -o jfr -f q19.jfr $(jps | awk '/surefirebooter/{print $1}')  # after ~25s warmup
jfrconv --cpu -o collapsed q19.jfr q19.collapsed
jfrconv --cpu -o html q19.jfr q19.html
```
